Deferred Content​

info

Deferred content is still not entirely nailed down and thus considered experimental.

The concept of deferred components represents another stage of templating: tayloring content to the specific recipient it's sent to. If this information is already available at the time of interpretation, the recipient may be passed to the call of the interpreter, causing deferred components to render immediately, as they now have access to the recipient-reference; otherwise, the result will contain one or more instances of a custom derivative of the platform's native component which essentially contains a context-closure that calls back into the deferred node once said reference is provided later on, right before sending/applying the component.

This concept solves the entire headache of server-side rendered (SSR) components, which simplifies platform-implementations by a great deal without having to compromise on available features; quite the opposite - we may introduce whichever SSR-tags and features that prove beneficial in a version-agnostic manner. Usually, these include: NBT-data, selectors, score-values and the like.

Advantages​

The main advantage deferred content has over simply employing environment-variables to interpret ASTs on a per-recipient basis is that this way, non-tailored components are constructed only once, instead of over and over again, yielding the exact same result each time; think of it as a partially cached tree. To ensure further efficiency, the paths to deferred components within the result are stored on a type of prefix-tree, avoiding to needlessly walk all children; once deferred components are invoked, their result is spliced into the final output by cloning along the necessary paths, leaving the immutable template untouched.
